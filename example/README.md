# recurrence_kit example

A single-screen demo app that exercises all three layers of the `recurrence_kit` package: the picker widget, the rule model, and the computation engine.

## Setup

From the example directory:

```bash
cd recurrence_kit/example
flutter pub get
flutter run
```

If the platform directories (`ios/`, `android/`, etc.) are missing after a fresh clone, regenerate them:

```bash
flutter create .
```

## What the app shows

The screen has four sections, all updating live as you interact with the picker:

**RecurrencePicker** — the full picker widget embedded in a card. Build any recurrence rule interactively: frequency, interval, day/week/month selection, and end conditions.

**RecurrenceRule.displayText** — the human-readable summary generated by the rule model. Updates on every picker change to show what `displayText` produces for the current configuration.

**Next occurrences** — the next 5 upcoming dates computed by `RecurrenceEngine.nextOccurrences`. If today matches the rule it appears as the first entry. When an "ends after N" condition is set, the count is respected (the example resolves it via `computeEndDateFromCount`).

**RecurrenceRule.toJson()** — the rule's JSON serialization, pretty-printed and selectable. Useful as a reference for consumers planning persistence or Drift integration.

## Theme toggle

The sun/moon icon in the app bar switches between two `RecurrencePickerTheme` configurations: a default light theme and a dark theme with a green accent. The picker card background also swaps to demonstrate how theming works in practice.

## Key implementation patterns

The example demonstrates several patterns that consumers of the package will likely need:

**Resolving "ends after N" rules.** The picker outputs `endAfterCount` but not a concrete `endDate`. The example shows the recommended workflow: call `RecurrenceEngine.computeEndDateFromCount` to resolve the end date before passing the rule to `nextOccurrences`. See the `_recompute` method.

**Keeping computation out of `build`.** Occurrence computation is cached in state fields (`_upcoming`, `_jsonString`) and recomputed only when the rule changes, not on every frame rebuild. This avoids lag when rapidly tapping stepper controls.

**Inclusive start date.** `nextOccurrences` treats `afterDate` as exclusive. The example passes `startDate - 1 day` as `afterDate` so that today's occurrence (if any) is included in the results.
